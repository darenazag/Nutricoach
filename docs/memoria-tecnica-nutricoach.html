1. Resumen ejecutivo

NutriCoach es una aplicacion full stack orientada a ayudar al usuario a registrar comidas, consultar su perfil nutricional, visualizar un dashboard de progreso y apoyarse en funciones de inteligencia artificial para analizar platos, conversar con un asistente nutricional y generar menus orientativos.

El estado auditado corresponde a una integracion avanzada del MVP. El producto combina una base funcional P0 con autenticacion, perfiles, comidas y dashboard en PostgreSQL, y un modulo IA persistido en MongoDB. La IA se conecta al flujo de usuario mediante una pantalla de asistente, un laboratorio tecnico, analisis de imagen en el registro de comida y guardado de platos analizados en el dashboard.

Estado de calidad verificado: backend lint OK, backend tests OK con 101 tests en 11 ficheros, frontend lint OK con 2 warnings no bloqueantes en MenuSugerido, y frontend build OK. La build informa de un warning no bloqueante por bundle JavaScript superior a 500 kB.

4. Arquitectura general

La arquitectura se organiza como una SPA React que consume un backend Express monolitico. Dentro del backend conviven dos dominios: el core funcional P0 y el modulo IA. El core P0 usa PostgreSQL; el modulo IA usa MongoDB para datos semiestructurados y auditables. 

Frontend React/Vite
  |-- paginas publicas: landing, login, registro
  |-- paginas privadas: perfil, registrar comida, editar perfil, asistente IA
  |-- servicios: api.ts para P0, aiApi.ts para IA/lab
        |
        v
Backend Express/TypeScript
  |-- /api/auth, /api/profile, /api/meals, /api/foods, /api/users
  |-- /api/ai/*
        |
        +--> PostgreSQL: User, Profile, Meal, Food_item, Profile_Meal
        |
        +--> MongoDB: AiConversation, AiMessage, AiPlateAnalysis, AiCacheEntry, AiWeeklyMenuPlan
        |
        +--> Proveedores IA: DeepSeek y Gemini

6.4 Modelo SQL

El esquema SQL principal define las tablas:

  • User: usuario, email y password.
  • Profile: peso, edad, altura, sexo, actividad, objetivo, TMB y TDEE.
  • Meal: comida preparada con calorias y macronutrientes.
  • Food_item: alimentos base.
  • Meal_Food_item: relacion N:M entre comidas y alimentos.
  • Profile_Meal: comidas asignadas a un perfil.
Riesgo tecnico: el esquema usa IDs numericos manuales y no secuencias/identity. Esto obliga a generar IDs en aplicacion o scripts y puede crear colisiones si el flujo crece.

7. Modulo IA

7.1 Objetivo

El modulo IA aporta capacidades de lenguaje natural y vision sin sustituir la logica P0. Su diseno se basa en recibir contexto normalizado, validar la entrada, renderizar prompts versionados, llamar al proveedor IA, validar la salida y persistir resultados auditables.

7.4 Proveedores IA

  • DeepSeek: proveedor primario para texto si AI_ENABLE_DEEPSEEK=true y AI_TEXT_PROVIDER=deepseek.
  • Gemini: proveedor multimodal para imagen y fallback para texto cuando DeepSeek falla o no valida JSON.
  • Las claves reales no se almacenan en el repositorio; se cargan desde .env.

7.5 Seguridad en IA

  • aiRouter.use(authenticate) protege todos los endpoints IA.
  • El userId se deriva de req.auth.sub; el backend no confia en el userId enviado por cliente.
  • La lectura de conversaciones aplica ownership y evita filtrar conversaciones ajenas.
  • Los prompts incluyen reglas de seguridad nutricional y las respuestas contienen estructura safety.

7.6 Flujo de analisis de plato

Usuario sube imagen en RegistrarComida
  -> frontend envia multipart con JWT
  -> multer valida campo image y tamano
  -> sharp lee metadata y redimensiona si es necesario
  -> adapter obtiene contexto del perfil P0
  -> runAiPlateAnalysis valida request con Zod
  -> cache lookup por hash de imagen + prompt + modelo
  -> Gemini analiza imagen si no hay cache
  -> Zod valida respuesta estructurada
  -> Mongo guarda AiPlateAnalysis y mensajes
  -> usuario revisa datos y guarda
  -> /api/ai/save-analyzed-meal escribe Meal + Profile_Meal en PostgreSQL
  -> dashboard lee la comida desde endpoints P0

11. Riesgos y deuda tecnica

Riesgo/deudaImpactoPrioridadRecomendacion
POST /api/meals exige admin. El flujo de usuario normal necesita /api/ai/save-analyzed-meal como workaround. Alta Revisar contrato P0 para permitir que un usuario autenticado cree sus propias comidas.
Backend depende de Mongo al arrancar. Si MONGO_URI falla, no se sirve ni la API P0. Alta Decidir si IA debe degradar sin bloquear P0, o documentar Mongo como dependencia obligatoria.
No hay rate limiting IA. Riesgo de coste y abuso con tokens validos. Alta Limitar por usuario/IP y por plan free/pro.
Prompts runtime desde memoria/defaults. El seeder Mongo existe, pero la edicion activa de prompts queda limitada. Media Cargar plantilla activa desde Mongo con fallback a defaults.
IDs SQL numericos no autogenerados. Riesgo de colisiones y complejidad en altas. Media Usar identity/serial o una estrategia centralizada.
Bundle frontend grande. Carga inicial mas pesada. Media Code splitting por rutas pesadas, especialmente AI Lab.
Documentacion historica no siempre refleja el stack real. Confusion en defensa tecnica o onboarding. Media Actualizar README general: React 19, pg, Mongo IA, endpoints actuales.
pgAdmin reiniciando en entorno local auditado. No bloquea app, pero afecta experiencia de demo/administracion. Baja Revisar credenciales/volumen de pgAdmin o retirarlo si no se usa.

12. Propuesta de roadmap post-MVP

  1. Corregir contrato P0 de creacion de comidas por usuario autenticado.
  2. Implementar rate limiting y control de coste IA.
  3. Unificar Docker demo: frontend, backend, PostgreSQL, MongoDB y health checks reales.
  4. Cargar prompts activos desde Mongo y crear panel/admin basico de prompts.
  5. Anadir CI minimo para lint, test y build en pull requests.
  6. Aplicar code splitting a rutas IA y AI Lab.
  7. Actualizar README y guia de despliegue para que reflejen el stack real.
  8. Instrumentar tokens/coste por conversacion y usuario.
  9. Mejorar robustez JSON de DeepSeek en menu semanal.
  10. Evaluar fase 2 nutricional con FatSecret o Edamam si el proyecto necesita datos verificados comerciales.

13. Conclusiones

NutriCoach presenta un MVP tecnicamente defendible: cuenta con frontend funcional, backend tipado, persistencia SQL, modulo IA integrado, validaciones estrictas, autenticacion por JWT y una suite de pruebas backend significativa. El modulo IA no esta aislado como demo separada: ya esta conectado con flujos de producto como registro de comida, dashboard y asistente.

La mayor fortaleza tecnica es la separacion entre datos funcionales y datos IA: PostgreSQL sigue siendo la fuente de verdad del negocio, mientras MongoDB absorbe conversaciones, cache y salidas semiestructuradas. Esta decision reduce el acoplamiento y facilita evolucionar el modulo IA.

Los principales puntos a reforzar antes de una entrega final o despliegue real son el contrato de creacion de comidas, el control de coste/rate limit, la unificacion de Docker/README y la reduccion del bundle frontend. Ninguno de estos puntos invalida el MVP, pero si marcan el camino para convertirlo en una version mas robusta y mantenible.
